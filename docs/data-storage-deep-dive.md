# Data Storage Deep Dive

Ecosystem survey, architecture evaluation, and recommendations for long-term storage of multi-state legislative vote data.

## Scope

This review covers the current tallgrass data inventory (Kansas, 1999-2026), projected multi-state growth (50 states × 25+ years), available storage technologies (CSV, Parquet, DuckDB, SQLite, PostgreSQL), and integration requirements for both analytical workloads and potential web applications.

The pattern is clear: **research-oriented projects use flat files; web-facing projects use PostgreSQL.** Projects that serve both audiences maintain both — flat files for researchers and a database for the web UI.

### Technology Comparison

#### CSV (Current)

**Strengths**: Human-readable, universal compatibility, zero dependencies, git-friendly diffs, trivial to share. Polars reads 633K rows in ~200ms. Every tool in every language can consume CSVs.

**Weaknesses**: No schema enforcement, no compression, no column pruning (must read entire file), string-typed (dates, booleans need parsing at load time), no indexing.

**Verdict**: Adequate for current Kansas-only scale. Becomes painful at 50-state scale — 10 GB of CSVs with no schema enforcement invites drift.

#### Parquet

**Strengths**: Columnar compression (10-20x over CSV for vote data), schema embedded in file, column pruning (read only needed columns), partitioning support, zero-server (just files), native Polars/Arrow support. A 85 MB votes CSV becomes ~5 MB Parquet.

**Weaknesses**: Binary format (not human-readable, not git-diffable), requires a reader library, no concurrent writes, no indexing beyond row groups.

**Verdict**: The natural evolution from CSV for analytical workloads. Polars reads Parquet ~5x faster than CSV due to column pruning and no parsing overhead. Partitioned by `(state, biennium)` mirrors our current directory structure.

#### DuckDB

**Strengths**: Embedded analytical database (no server), reads Parquet/CSV directly without import, SQL interface, columnar storage, window functions for time-series analysis, ACID transactions. Can query across partitioned Parquet files as a single table. Excellent Polars interop via Arrow.

**Weaknesses**: Single-writer (no concurrent writes from multiple processes), not designed for OLTP workloads, limited ecosystem for web frameworks, relatively young project (v1.x since 2024).

**Verdict**: Ideal as a query engine over Parquet files. Not a replacement for PostgreSQL in web applications, but can serve as the analytical layer alongside it.

#### SQLite

**Strengths**: Single-file database, zero-server, ACID, widely deployed, mature tooling. Good for read-heavy workloads.

**Weaknesses**: Single-writer lock (writes serialize globally), limited concurrent read performance under heavy load, no native columnar operations, type system is "flexible" (stores anything in any column), no built-in compression, limited analytical functions compared to DuckDB.

**Verdict**: Better than CSV for schema enforcement but worse than both DuckDB (analytics) and PostgreSQL (web serving). Falls in an awkward middle ground for this project.

#### PostgreSQL

**Strengths**: Full ACID compliance, concurrent reads and writes, mature indexing (B-tree, GIN, GiST), JSON support, excellent Django integration (first-class ORM support), proven at scale (handles billions of rows), rich extension ecosystem (PostGIS, pg_trgm, pgvector), decades of production hardening, `COPY` bulk loading, materialized views for expensive aggregations.

**Weaknesses**: Requires a running server (ops overhead), heavier than embedded solutions for pure analytical workloads, columnar operations require extensions (citus, timescaledb), storage is row-oriented (less efficient for wide scans over few columns).

**Verdict**: The right choice if a web application is in the picture. Django's ORM, migrations, and admin interface are built for PostgreSQL. For analytical queries, PostgreSQL handles 30M rows comfortably — it is not the bottleneck.

## Architecture Options

### Option A: CSV Status Quo

```
data/kansas/91st_2025-2026/91st_2025-2026_votes.csv
data/kansas/91st_2025-2026/91st_2025-2026_rollcalls.csv
...
```

No changes. Analysis scripts continue loading CSVs via Polars. Works today, becomes unwieldy at multi-state scale.

**Best for**: Single-state research, maximum simplicity.

### Option B: PostgreSQL + Django

```
database: tallgrass (PostgreSQL 16+)
tables: votes, rollcalls, legislators, bill_actions, bill_texts
         sessions, states (reference tables)
```

All data in PostgreSQL. Django models map to tables. Admin interface for data browsing. REST API (DRF) or GraphQL for external consumers. Scrapers write directly to the database via Django ORM or raw `COPY`.

**Best for**: Web application, API-backed services, collaborative access.

**Best for**: Full platform — web UI, API, analytical pipeline, and multi-state scaling. The recommended option.

## PostgreSQL Schema Design

If PostgreSQL is adopted (Options C or D), the schema follows Django conventions while preserving the current data model.

### Core Tables

```sql
-- Reference tables
CREATE TABLE states (
    code        CHAR(2) PRIMARY KEY,  -- 'KS'
    name        TEXT NOT NULL,         -- 'Kansas'
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE sessions (
    id              SERIAL PRIMARY KEY,
    state_code      CHAR(2) REFERENCES states(code),
    start_year      INT NOT NULL,
    end_year        INT NOT NULL,
    legislature_num INT NOT NULL,       -- 91
    label           TEXT NOT NULL,       -- '91st 2025-2026'
    is_special      BOOLEAN DEFAULT FALSE,
    UNIQUE (state_code, start_year, is_special)
);

-- Data tables (match current CSV structure)
CREATE TABLE legislators (
    id          SERIAL PRIMARY KEY,
    session_id  INT REFERENCES sessions(id),
    slug        TEXT NOT NULL,           -- 'sen_masterson_ty_1'
    name        TEXT NOT NULL,
    full_name   TEXT,
    chamber     TEXT NOT NULL,           -- 'Senate' | 'House'
    party       TEXT NOT NULL DEFAULT '',
    district    TEXT,
    member_url  TEXT,
    ocd_id      TEXT,                    -- OpenStates person ID
    UNIQUE (session_id, slug)
);

CREATE TABLE rollcalls (
    id              SERIAL PRIMARY KEY,
    session_id      INT REFERENCES sessions(id),
    vote_id         TEXT NOT NULL,       -- 'je_20250320203513' or 'kf_33_2011_S'
    bill_number     TEXT NOT NULL,
    vote_datetime   TIMESTAMPTZ,
    vote_date       TEXT,                -- original format preserved
    chamber         TEXT NOT NULL,
    motion          TEXT,
    passed          BOOLEAN,
    vote_type       TEXT,
    yea_count       INT,
    nay_count       INT,
    pnp_count       INT,
    anv_count       INT,
    nv_count        INT,
    sponsor_slugs   TEXT,
    source_url      TEXT,
    UNIQUE (session_id, vote_id)
);

CREATE TABLE votes (
    id              BIGSERIAL PRIMARY KEY,
    rollcall_id     INT REFERENCES rollcalls(id),
    legislator_id   INT REFERENCES legislators(id),
    vote_category   TEXT NOT NULL,       -- 5 canonical categories
    UNIQUE (rollcall_id, legislator_id)
);

-- Indexes for common query patterns
CREATE INDEX idx_votes_legislator ON votes(legislator_id);
CREATE INDEX idx_rollcalls_bill ON rollcalls(bill_number);
CREATE INDEX idx_rollcalls_chamber ON rollcalls(session_id, chamber);
CREATE INDEX idx_legislators_party ON legislators(session_id, party);
CREATE INDEX idx_legislators_ocd ON legislators(ocd_id) WHERE ocd_id IS NOT NULL;
```

### Bulk Loading

PostgreSQL's `COPY` command loads CSVs at ~500K rows/second. Our entire 633K-row votes table loads in ~1.3 seconds.

## Why Not Parquet?

Parquet is a columnar binary format that excels at compressing and scanning terabyte-scale analytical datasets. For tallgrass, the honest assessment:

| Parquet Advantage | Does It Matter Here? |
|-------------------|---------------------|
| 10-20x compression | 85 MB → ~5 MB, but 85 MB is already trivial |
| Column pruning (read only needed cols) | Polars reads full CSV in ~200ms — not a bottleneck |
| Schema embedded in file | PostgreSQL migrations give stronger enforcement |
| Faster Polars reads (~40ms vs ~200ms) | MCMC runs take 7 minutes — 160ms savings is noise |

**Parquet adds complexity without solving a real problem at our scale.** The projected 50-state dataset (~10 GB CSV) is still comfortably within PostgreSQL's range. Maintaining a separate Parquet export layer (keeping it in sync with PostgreSQL) adds operational burden for marginal performance gains.

If analytical query performance ever becomes a bottleneck (unlikely before hundreds of millions of rows), DuckDB can query PostgreSQL directly via its `postgres_scanner` extension — no Parquet intermediate needed.

**Bottom line**: CSV is human-readable, git-diffable, and universally compatible. PostgreSQL is queryable, schema-enforced, and web-ready. Adding a third format (Parquet) creates sync complexity that our data volume does not justify.

## Migration Path

A two-phase path that keeps the CSV workflow intact while building toward PostgreSQL.

### Phase 1: CSV Status Quo (Now)

No changes. Scrapers write CSVs. Analysis pipeline reads CSVs via Polars. This continues working at every stage and serves as the fallback.

### Phase 2: PostgreSQL + Django (When Web App is Scoped)

Set up PostgreSQL and Django. Load existing CSVs into the database. The scraper continues writing CSVs; a post-scrape step loads into PostgreSQL.

```bash
just scrape 2025               # writes CSVs (unchanged)
just db-load 2025-26           # CSV → PostgreSQL via COPY
```

The analysis pipeline can read from either source — Polars reads CSVs directly, or queries PostgreSQL via `pl.read_database()`. Both work.

### Phase 3: Direct Database Writes (Mature)

Once Django models are stable and trusted, scrapers write directly to PostgreSQL. CSVs become an export format for distribution and backup.

```bash
just scrape 2025               # writes to PostgreSQL directly
just export --format csv       # PostgreSQL → CSV (for sharing/backup)
```

## Recommendation

**Option C (PostgreSQL + Django)** is the right long-term architecture, given:

1. **PostgreSQL expertise** — the team already uses PostgreSQL extensively
2. **Django integration** — web application with admin interface, REST API, migrations
3. **Multi-state scaling** — PostgreSQL handles the projected 30M-row vote table comfortably
4. **Human readability** — `psql` + Django admin provide searchable, browsable access
5. **Schema enforcement** — Django migrations prevent drift as states are added
6. **CSV export** — `COPY TO` produces CSVs for researchers who don't need a database

**Start with Phase 1** (CSV status quo) — zero changes needed. Move to Phase 2 when a Django application is scoped. CSV workflow continues as fallback and export format at every stage.

### What Not to Do

- **Do not add Parquet as an intermediate layer.** It adds sync complexity for performance gains our dataset does not need. If analytical performance ever becomes a bottleneck, DuckDB can query PostgreSQL directly.
- **Do not adopt DuckDB as the primary store.** It is single-writer and has no web framework ecosystem.
- **Do not adopt SQLite.** It occupies an awkward middle ground — worse than PostgreSQL for web serving, worse than DuckDB for analytics.
- **Do not stop writing CSVs until PostgreSQL is the proven authoritative source.** CSVs are the safest fallback — human-readable, git-diffable, universally compatible.
- **Do not load MCMC posteriors into a database.** The 35 GB of NetCDF files are ephemeral computation artifacts. They belong on the filesystem.
